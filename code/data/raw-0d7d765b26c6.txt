inside a separate OS process and there is no IPC op that returns it, so it never
enters the orchestrator process; the tests spawn an actual child and verify a

shipped the stub host to production as-is, the spawner would control verdicts, which
would be a worse trust hole than the one #2 closed. So: custody (key isolation) is
genuinely closed; deliberation-trust (the model deciding child-side) is the explicit
remaining testnet step, and the review context says so plainly. Closed-in-code,
honestly scoped.

#2 state-log — a correct, minimal interim: the same hash-chain that makes votes
tamper-evident, applied to module state transitions, with a demo proving a silent
slash->release rewrite is caught at the edited entry. Two honest limits, both
disclosed: (a) it is the MECHANISM — the modules do not yet auto-emit into it, so
today it is exercised by the demo, not wired through bonds/lifecycle/etc.; (b) it is
tamper-EVIDENCE only — per-event authorization (who signed off) and real on-chain
anchoring are testnet. Neither is overclaimed. As an interim per CIP-3/CIP-4 §5,
this is the right shape.

Reasoning: all three do what they claim; no regression; no new intent-vs-check gap
