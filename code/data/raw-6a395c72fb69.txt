## Assessment against each claim

**1. NO_VERDICT never ratifies** — `ratify()` skips NO_VERDICT in the max-finding loop (`signed-vote.ts:187`) and gates `ratified = verdict !== null && max >= required` (`:194`). NO_VERDICT stays in `tally` for transparency but can never *be* the verdict. A neighboring-position check confirms a real verdict below threshold (e.g. REVISE:1 vs NO_VERDICT:2, required 2) correctly yields `ratified:false`. Backed by a direct test. ✅

**2. Real-verdict participation** — daemon computes `realVotes = votes.filter(v => v.verdict !== 'NO_VERDICT')` and finalizes only at `realVotes >= quorum` (`daemon.ts:62-63`); otherwise retry/fail. Crucially it filters the `votes` array itself, **not** `ratify().counted`/`.tally` (which include NO_VERDICT voters) — so the laundering path is closed at the daemon layer too. Verified `convene()` returns the same signed-vote set `ratify()` tallies (`panel.ts:95-96`), and that an errored invoker still yields a *signed* `NO_VERDICT` vote (`deliberating-signer-host.ts:50-53`), so the two layers see a consistent view. ✅

**3. Full-sha dedup** — id is `review-${epoch}-${commit.sha}` (full sha); short 12-char is display-only. The collision test (shared epoch + 12-char prefix, distinct full sha → distinct ids) passes. Strengthens tamper-evidence. ✅

**4. Invoker budgets** — 480s timeout / 16MB buffer shared, hermes `--max-turns 12`, with SIGTERM→"timed out after Nms" diagnostics. Prompt passed via env (`QRM_PROMPT`) for the shell-routed invokers / argv for hermes — no injection, custody untouched. ✅

## Adversarial findings (non-blocking)

- **Daemon uses `params.quorum` for the participation bar, not `max(quorum, supermajority(n))`.** For the standing 3-panel (quorum 2, supermajority ⌈6/3⌉=2) these coincide, so no live bug. And even under a misconfig where quorum < supermajority, `ratify()` *independently* enforces the 2/3 floor, so the `ratified` flag stays correct — the daemon's bar only governs retry-vs-finalize and errs conservatively (finalizes as *not* ratified, never laundering a NO into a YES). Not a soundness break.
- **`commons.ts:84-89` would label a NO_VERDICT position as `CREDIBLE_MINORITY` within a RESOLVED claim** — cosmetic/reputational display only, doesn't touch ratification or finalization, and is outside this commit's surface.

Custody, no-laundering, liveness, and tamper-evidence all hold; every claim is implemented as stated and covered by tests (TDD satisfied). The change does what it says with no gap between claim and implementation.
