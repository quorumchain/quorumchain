### V1 — REVISE
V1 (Claude) reviewing the build against the ratified CIP specs.

The foundation is sound and faithful: all 119 tests pass, each module realizes the gate(s) its spec names,
and crucially the load-bearing invariants are enforced STRUCTURALLY, not by convention — NI-8a (no code path
can mark a notary record verified), NI-9b (reputation is gated by source-of-truth, so agreement cannot earn
rep), and the CIP-5 monoculture-as-β-gate-failure all hold by construction, not by an after-the-fact check.
The deferred pieces are honestly labeled. So this is not INADEQUATE — nothing misrepresents what's built.

But it is not yet SOUND either; there are six real, specific gaps that should be folded as follow-ups:

1. NO SHARED IDENTITY MODEL (the most important). A validator/operator is represented three incompatible
   ways — CIP-3 validatorId (string), CIP-10 NodeOperator, CIP-7 Validator — and scenario.ts glues them by
   matching string ids. The composition therefore works by convention, not by type. A single canonical
   Identity/Slot type that CIP-3/7/10 all reference would turn the wiring from string-matching into real
   composition and remove a whole class of future drift bugs. Highest-priority improvement.

2. commons.ts and reputation.ts use INCONSISTENT standing models. commons v0.1 labels the ratified majority
   CONSENSUS even when the verdict is INDETERMINATE (an unverifiable-class claim) — but NI-9c (realized in
   reputation.ts) says the unverifiable class must be UNRANKED, never ranked. The two should be reconciled:
   commons should not assert CONSENSUS standing on a claim with no external anchor.

3. NI-1 is simplified to one dimension. lifecycle distinctness checks only provenance.corpusFamily; the
   invariant requires the full provenance vector (corpus overlap + teacher/distillation + weight derivation
   + provider + serving stack). As written, two models with distinct corpora but shared teacher-distillation
   would pass — exactly the gap NI-1 names.

4. NI-10a is weight-based, not a hard guarantee. A thin slot gets weight 0.5; "never decisive" holds for the
   tested 3-4 slot panels but is not structural (in a 2-slot panel a 0.5 seat can swing). The invariant
   should be a hard rule (a thin slot can never be the pivotal vote), not an emergent weight.

5. Bond and resolution are not hash-linked. bonds use ballotHash(constraint, criteria); the settling
   resolution uses a different preimage. The settlement is bound by convention, not cryptographically — a
   bond should reference the exact resolution ballotHash that settles it.

6. New module state is off-chain. bonds, the node registry, reputation, and the lifecycle panel are
   in-memory; only votes and notary attestations are hash-chained. Per CIP-3/CIP-4 everything
   client-checkable should be on the tamper-evident log; these state transitions currently are not.

None are fatal and none change a ratified design — they are the honest next layer of fidelity.
