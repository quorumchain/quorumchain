# Round 62 — CIP-17: Solana Memo External Anchoring (tamper-evidence beyond the host)

**Date:** 2026-06-07
**Type:** New improvement proposal — add an **external, third-party anchor of record** for the
Quorumchain signed log by periodically committing the chain's tip hash into a **Solana memo
transaction**, so that a rewrite of local history becomes **detectable against a substrate
Quorumchain does not control**. Design ruling (YES / NO), pending panel ratification on the
core chain.

## 1. Problem — append-only is self-attested up to the host boundary

Quorumchain's append-only history (T0) is enforced by a SHA-256 hash chain over Ed25519-signed
votes. That construction makes any rewrite **internally** detectable: change a past vote and
every subsequent `prevHash` mismatches, and the signatures fail. But the integrity proof is
**self-contained** — it lives entirely inside the same repository and on the same hosts that
hold the data. An actor with write access to the log (a compromised host, a coerced operator,
or a future maintainer) can in principle **re-sign a divergent history from a chosen fork
point** and present it as canonical, because nothing outside Quorumchain has ever **witnessed**
the original tip.

Concretely: the hash chain proves *internal consistency* (this log is a valid chain) but not
*temporal singularity* (this is the only chain that ever existed). Two internally-valid chains
that diverge at round N are indistinguishable to a reader who only has one of them. The missing
property is an **external, timestamped witness** to the tip hash that Quorumchain cannot
retroactively alter — an anchor of record on an independent substrate.

This is the same class of gap CIP-15 closed *inside* re-adjudication (a self-asserted anchor is
not an anchor): here the self-assertion is the chain attesting to its own past. The fix is to
periodically publish the tip hash where Quorumchain has no rewrite authority.

## 2. Proposed change (exactly what YES enacts)

**2.1 — Periodic tip-hash commitment to a Solana memo transaction.** At a defined cadence,
Quorumchain submits a transaction carrying, in the SPL **Memo program**, a commitment over the
current core-chain tip (the latest `ballotHash` / log head and a monotonic anchor sequence
number). The transaction's confirmation on Solana becomes a timestamped, externally-witnessed
record of "Quorumchain's tip was X at Solana slot S." Solana is chosen as a high-availability,
low-cost public ledger purely as an **anchor of record** — Quorumchain's consensus remains
entirely the AI panel; Solana never adjudicates anything.

**2.2 — Three layers (A / B / C).**
- **Layer A — the source chain (unchanged).** The existing Ed25519-signed, SHA-256 hash-chained
  `votes.log` is and remains the authoritative chain. CIP-17 adds **no** dependency of Layer A
  on Solana: the panel, ratification, and `verifyLog` are byte-for-byte unchanged. If Solana is
  unreachable, Layer A keeps operating normally.
- **Layer B — the in-repo tip-commitment chain.** A separate, append-only in-repo record
  (`anchors.jsonl`-class) where each entry binds `{anchorSeq, tipHash, solanaTxSig, slot,
  asOf}` into its own hash chain. Layer B is the local, version-controlled mirror of what was
  anchored externally; it lets a verifier reconstruct the anchor history offline and cross-check
  it against Solana. Layer B is itself append-only and forward-only.
- **Layer C — the external witness (Solana mainnet-beta).** The **anchor of record is the
  mainnet-beta memo transaction.** Devnet is used for tests/CI only and is never treated as an
  anchor of record. Each Layer-B entry references the Layer-C transaction signature that
  witnessed it.

**2.3 — Isolated dependency surface.** All `@solana/web3.js` usage is confined to a single new
module `code/src/solana-anchor.ts` (submit-memo, fetch-confirmation, parse-memo). The CIP-15
`code/src/anchor.ts` (the Tier-1 re-adjudication anchor verifier) is a **distinct concern and
is kept separate** — CIP-17 does not touch it, import it, or change its semantics. No Solana
type or import leaks into `panel.ts`, `vote-log.ts`, `commons.ts`, or `anchor.ts`.

**2.4 — A `verify-anchored` path.** Given the local Layer-A log and Layer-B commitment chain, a
verifier can (offline-then-online) confirm: each Layer-B entry's `tipHash` matches a real Layer-A
prefix tip; each Layer-B entry references a Solana mainnet-beta transaction whose memo carries
exactly that commitment; and the Layer-B chain is internally consistent. A local rewrite that
diverges after the most recent anchored tip is then **detectable** — the rewritten chain cannot
produce a Solana-witnessed commitment for its divergent tip without forging a confirmed
mainnet-beta transaction.

**2.5 — Low-privilege anchoring key, distinct from validator keys.** Submitting a memo
transaction requires a Solana keypair that funds the (tiny) transaction fee. This key is a
**dedicated, low-privilege anchoring key** whose **only** capability is paying for and signing
memo submissions. It is **NOT** any validator's Ed25519 signing key, holds no consensus
authority, and its compromise can at worst cause spurious/withheld anchors — it can **never**
sign a vote, ratify a ballot, or alter Layer A. The anchoring key is operationally and
cryptographically separated from the keystore that holds V1/V2/V3 keys.

## 3. Affected invariants — new NI-17a, NI-17b, NI-17c, NI-17d

- **NI-17a (Layer-A sovereignty — unchanged).** Quorumchain consensus, ratification, and the
  authoritative hash-chained log are produced **solely** by the AI panel. Solana is an
  **anchor of record only** and has **no** consensus, adjudication, or veto role. No protocol
  decision is ever gated on Solana availability; Layer A operates unchanged when Layer C is
  unreachable. *(This invariant restates and protects the T0 separation-of-powers and
  diversity-as-principle core — external anchoring may witness, never govern.)*

- **NI-17b (Solana anchor obligation).** When external anchoring is enabled, the current tip
  hash MUST be committed to a confirmed **mainnet-beta** memo transaction at the defined cadence,
  and the corresponding Layer-B entry MUST bind `{anchorSeq, tipHash, solanaTxSig, slot, asOf}`
  in the append-only Layer-B chain. Devnet confirmations satisfy tests/CI but NEVER count as an
  anchor of record.

- **NI-17c (rewrite-detectability).** Any rewrite of Layer-A history that diverges at or before
  the most recently anchored tip is detectable via `verify-anchored`: the divergent chain cannot
  exhibit a confirmed mainnet-beta memo commitment for its tip without forging a Solana
  transaction. The anchor strengthens (never weakens) the existing internal hash-chain
  guarantee.

- **NI-17d (honest pre-anchor boundary).** CIP-17 makes **no** claim of detectability for
  rewrites confined to the **unanchored suffix** — i.e. log entries newer than the most recent
  confirmed anchor. The protection is explicitly bounded to history **at or before** the latest
  anchored tip; the unanchored tail remains protected only by the internal hash chain, exactly
  as today. The cadence sets the size of this honest boundary, and the bound is stated rather
  than overclaimed.

All four are **additive**: NI-17a/b/c/d introduce a new external-witness mechanism and a new
in-repo Layer-B chain. **No T0 frozen-unamendable core item** (append-only history,
diversity-as-principle, separation-of-powers, frozen tier-assignment function) is amended —
NI-17a is a guard that *reinforces* separation-of-powers, not a relaxation of it. **No prior CIP
invariant is amended:** Layer A and the CIP-15 anchor verifier are untouched; CIP-17 adds layers
in front of/beside them.

## 4. Compatibility / migration

- **Append-only / forward-only:** preserved on both chains. Layer A is not mutated at all. Layer
  B is a **new** append-only hash chain; anchoring **appends**, never rewrites.
- **No replay/verify change to existing data:** the 169+ existing convenings and every signed
  vote verify byte-identically — CIP-17 reads the tip, it never re-hashes or re-binds a past
  vote. `verifyLog` is unchanged.
- **Optional / degradable:** when Layer C is unreachable or anchoring is disabled, Layer A
  proceeds normally and `verify-anchored` reports the chain as internally valid with the anchor
  coverage it has (up to the last confirmed anchor). Solana downtime never halts consensus
  (NI-17a).
- **Dependency isolation:** `@solana/web3.js` enters only via `solana-anchor.ts`; the rest of the
  codebase has no Solana import surface, keeping the substrate swappable and the blast radius of
  a Solana-side issue contained to the anchoring module.

## 5. Sovereignty & key tradeoffs (named honestly)

- **A new external dependency.** Anchoring introduces a soft dependency on a substrate
  Quorumchain does not control. CIP-17 bounds this: the dependency is **witness-only** (NI-17a),
  never load-bearing for consensus, and degrades cleanly. The honest cost is that the *strength*
  of external tamper-evidence now partly rests on Solana mainnet-beta's continued existence and
  liveness — but the *correctness* of Quorumchain never does.
- **A new key in the trust surface.** The low-privilege anchoring key is a new secret to manage.
  CIP-17 bounds its blast radius (NI-17a + §2.5): it can only pay for / sign memo submissions, so
  its worst-case compromise is spurious or withheld anchors — a *liveness/noise* failure on Layer
  C, never an *integrity* failure on Layer A. It is cryptographically and operationally distinct
  from the validator keystore.
- **Honest boundary, not a total proof.** NI-17d states plainly that the unanchored suffix is not
  externally witnessed. CIP-17 buys detectability for anchored history, not for the in-flight
  tail; we name the boundary rather than imply a stronger guarantee than the cadence supports.

## 6. Verdict criteria

**Single conclusion: should Quorumchain enact CIP-17 (Solana memo external anchoring) as
specified above? Yes or no.**

A **YES** verdict:
1. enacts **NI-17a** (Layer-A sovereignty), **NI-17b** (Solana anchor obligation, mainnet-beta as
   anchor of record), **NI-17c** (rewrite-detectability), and **NI-17d** (honest pre-anchor
   boundary);
2. enacts the **three-layer design** — Layer A unchanged, Layer B in-repo append-only
   tip-commitment chain, Layer C the mainnet-beta memo witness (devnet for tests only);
3. authorizes the isolated `code/src/solana-anchor.ts` module (sole `@solana/web3.js` surface),
   the `verify-anchored` path, and the **low-privilege anchoring key distinct from the validator
   keys** (§2.5);
4. keeps the CIP-15 `anchor.ts` re-adjudication verifier **separate and untouched**.

A **NO** verdict must name at least one of:
- a **sovereignty / Solana-dependency objection** — why witness-only external anchoring is
  unacceptable for Quorumchain — **together with a named alternative** substrate or mechanism;
  or
- a **rewrite that passes `verify-anchored`** — a concrete divergence at/before the latest
  anchored tip that the design fails to detect (i.e. NI-17c does not hold); or
- a **new-key risk the §2.5 bounding misses** — a capability the low-privilege anchoring key
  could exercise that reaches Layer A / consensus despite the stated separation; or
- **superseded-by-substrate** — a reason the in-repo hash chain (or an existing CIP) already
  provides this property such that external anchoring is redundant.

## 7. Test plan (TDD — written and shown failing/passing before merge, at implementation time)

1. **Layer-B commitment binds the real tip.** A Layer-B entry whose `tipHash` does not match the
   Layer-A prefix tip is rejected by `verify-anchored`.
2. **Rewrite-detectability (NI-17c).** A log rewritten before the latest anchored tip fails
   `verify-anchored` because no confirmed memo witnesses its divergent tip; PASSES (verifies)
   only for the genuine anchored chain.
3. **Honest boundary (NI-17d).** A rewrite confined to the unanchored suffix is *not* claimed
   detectable — the test asserts the bound is reported honestly, not overclaimed.
4. **Sovereignty / degradation (NI-17a).** With Layer C unreachable, Layer A consensus and
   `verifyLog` proceed unchanged and `verify-anchored` reports coverage up to the last anchor.
5. **Key separation (§2.5).** The anchoring key cannot sign a vote / ratify a ballot; only
   validator keystore keys can — asserted structurally.
6. **Dependency isolation.** `@solana/web3.js` is imported only by `solana-anchor.ts`; no Solana
   import surface in `panel.ts` / `vote-log.ts` / `commons.ts` / `anchor.ts`.
7. **Mainnet-vs-devnet (NI-17b).** A devnet confirmation never counts as an anchor of record;
   only a confirmed mainnet-beta memo does.

---

**Standing lesson:** the hash chain proves a log is *self-consistent*; it cannot prove the log is
*the only one that ever existed*. CIP-17 borrows exactly enough from an external substrate to
witness the tip — and no more. The design is deliberately the narrowest external dependency that
buys temporal singularity: witness-only (never governing, per NI-17a), bounded to anchored
history (NI-17d states the boundary instead of hiding it), isolated to one module, and keyed by a
low-privilege secret that can only ever produce noise, never an integrity failure. External
anchoring may witness the chain; it may never govern it.
