# Round 61 — CIP-16: Distinct-Question Separator (lineage grouping key)

**Date:** 2026-06-06
**Type:** New improvement proposal — a corrective amendment to the CIP-13 supersession
lineage so that two genuinely **distinct questions** that happen to cite the **same
supersede-root** are projected as **separate sibling lineages**, not silently merged into
one. Design ruling (YES / NO), pending panel ratification on the core chain.
**Status:** PROPOSED. Implementation, the #38 un-merge migration, and the #38-factual
re-convene are all gated behind a 2/3 YES on this round's single conclusion.

---

## 1. Problem — the #38 case (concrete evidence)

The pre-registry Barron-Trump / DJT-memecoin convening (#38, original prompt/context
cryptographically unrecoverable; ballot root `b644657f7fc859712014083c295335b4917decd1e18af4ba4159f4dbbc569830`) actually
adjudicated **two distinct questions**:

- **#38-factual** (`a611849bfe3c1c60eda7ccf7cfd0d43c5b9b05ba7a06830a4c4dc176c113b84f`) —
  *Was Barron Trump involved in some way with the DJT memecoin, under a
  preponderance-of-evidence standard?* (NORMATIVE.)
- **#38-procedural** (`a9f6c5956578bd52c13531417827b2530e28871403c0f9a4eaff3c1decd7efae`) —
  *Was Polymarket's override of its own NO-resolving UMA oracle a legitimate
  resolution?* (NORMATIVE.)

When the CIP-9 registry statements were recorded, **both** were declared as
`supersedes: b644657…` — each is a faithful re-statement of one half of the same
original convening. But the CIP-13 lineage pass groups every ballot **solely by its
lineage root** (`rootOf(bh)`). With two members sharing root `b644657`, the projection
treats them as **competing versions of one claim**: the later one (by log order) is
selected as the lineage head and the earlier one is demoted to a `priorVersion` of a
question it never asked. The Commons then shows the *procedural* question superseding —
"replacing the older, now-historical version of" — the *factual* question, which is
simply false. Two different questions are merged into one lineage.

This is a projection bug, not a chain bug: every ballot, signature, and hash is intact
and verifiable. The merge is an artifact of an over-broad grouping key.

## 2. Proposed change (exactly what YES enacts)

**2.1 — A new optional declared meta field `meta.questionId`** (a short stable string
identifying *which question* a ballot adjudicates within a supersede chain; e.g.
`"38-factual"`, `"38-procedural"`). It is **declared meta in the same class as
`supersedes`**: advisory at declaration, recorded in the registry, projected by the read
path. It is **UNHASHED** — it is NOT added to `boundTypeOf` or `anchorCommitmentOf`, so
it does **not** enter `ballotHash` and the already-shipped #38a/#38b registry entries
continue to pass `verifyEntry` unchanged.

**2.2 — Change the lineage grouping key** in `code/src/commons.ts` from `rootOf(bh)` to
the composite `(rootOf(bh), questionId ?? '')`. Ballots that share a supersede-root but
declare *different* `questionId`s form **separate sibling lineages**; ballots that share
a root and an (absent-or-equal) `questionId` group exactly as they do today.

**2.3 — Within each sibling group, the head-selection and admission gates are byte-for-byte
unchanged.** The separator only partitions the member set *before* the existing per-group
logic runs. The CIP-15 split gate (`reviewAdmissible` / `promotable`), the CIP-13 NI-13h
type-invariance check, the NI-13e/NI-15b anchor gate, and the log-order head rule (NI-13g)
all run **per sibling group, exactly as before**.

## 3. Affected invariants — new NI-16a, NI-16b

- **NI-16a (separation).** Two ballots that share a CIP-13 supersede-root but declare
  distinct `questionId`s belong to **distinct lineages**. Neither is the other's
  `priorVersion`, `current`, or `pendingReview` entry. A ballot with no `questionId`
  groups under the empty-string separator (back-compatible with every pre-CIP-16 ballot).

- **NI-16b (separator splits groups, never relaxes the gate).** `questionId` may only
  **partition** a lineage group into smaller sibling groups. It can never **merge**
  groups, never promote a successor that the CIP-13/CIP-15 gate would reject, and never
  weaken the **CIP-13 NI-13h** type-invariance gate or the **CIP-15 NI-15b**
  halt-over-degrade supersede gate. Formally: for any partition the separator induces, the
  set of promotable successors is a **subset** of what the un-separated grouping would
  admit — separation can only *withhold* a promotion (by isolating a successor from the
  head it would have superseded), never *grant* one. The separator is monotone-downward on
  promotions.

Both new invariants are additive (new mechanism + a strictly-narrowing partition). No T0
frozen-unamendable core item (append-only history, diversity-as-principle,
separation-of-powers, frozen tier-assignment function) is touched. No prior CIP invariant
is amended or relaxed — CIP-13/14/15 are **preserved**; the change adds a partition in
front of them.

## 4. Compatibility / migration

- **Append-only / forward-only:** preserved. No existing entry, vote, or hash is mutated.
  The #38 un-merge is performed by **appending** new registry entries that re-state each
  sibling's existing prompt/context/meta (so `verifyEntry` still passes) and **add** the
  distinct `questionId` — exactly the append-only re-statement discipline `attachDossier`
  already uses.
- **Backward-compatible projection:** the 169 existing convenings carry no `questionId`,
  so every one groups under `questionId=''` and projects byte-identically to today
  (verified by the 421-test suite remaining green plus a dedicated back-compat test).
- **`questionId` stays unhashed:** because it is NOT in `boundTypeOf`/`anchorCommitmentOf`,
  the #38a (`a611849…`) and #38b (`a9f6c5…`) entries — already on the signed log — keep
  verifying. Adding it to the hash would break them; CIP-16 deliberately does not.

## 5. Root `b644657` disposition — PUT TO THE PANEL

The pre-registry root `b644657` (#38) has no registry entry of its own (its original
prompt/context are unrecoverable) and is the shared supersede-root of both siblings. Once
the siblings are separated into two lineages, the root must land in exactly one of the
following dispositions. **A YES verdict on this round SELECTS one; the proposal does not
pre-decide it.** State the selected option (A or B) in your verdict reasoning.

- **Option A — tag the root into the factual sibling.** Treat `b644657` as carrying
  `questionId="38-factual"`, so it heads the factual lineage and `a611849…` supersedes it
  there; the procedural sibling (`a9f6c5…`) is then a fresh lineage with no prior. Rationale:
  the original #38 convening's *recorded* verdict (INDETERMINATE 2/3, V3 dissent NO) was the
  factual-involvement question; the procedural override question is the genuinely new
  re-statement. Cost: requires recording a `questionId` for a root that has no registry entry
  (done as an appended meta-only re-statement keyed by ballotHash, no prompt recovery claimed).

- **Option B — leave the root a lone decomposed node.** `b644657` stays a stand-alone
  ancestor that decomposed into two new sibling questions; **both** `a611849…` (factual)
  and `a9f6c5…` (procedural) are heads of their own fresh lineages, each *referencing*
  `b644657` as the decomposed origin but neither inheriting the root as a prior version.
  Rationale: the original conflated both questions, so neither re-statement is strictly the
  "same question" the root asked; representing it as a decomposition is the most honest. Cost:
  the root shows as superseded-by-nothing rather than promoting into the factual line.

## 6. Verdict criteria

**Single conclusion: should Quorumchain enact CIP-16 as specified above? Yes or no.**

A **YES** verdict:
1. enacts **NI-16a** (separation) and **NI-16b** (separator splits, never relaxes the gate);
2. enacts the `(rootOf(bh), questionId ?? '')` grouping key and the optional unhashed
   `meta.questionId` field;
3. **authorizes the #38 un-merge migration** (append-only re-statements adding
   `questionId="38-factual"` to `a611849…` and `questionId="38-procedural"` to `a9f6c5…`);
4. **selects the root `b644657` disposition** — Option A (tag into factual sibling) **or**
   Option B (lone decomposed node) — stated in the verdict reasoning.

A **NO** verdict must name at least one of:
- a **merge-is-correct rationale** — why the two distinct questions *should* remain one
  lineage; or
- a **gate-relaxation path** — a concrete case where the separator could promote a
  successor the CIP-13 NI-13h / CIP-15 NI-15b gate would otherwise reject (i.e. NI-16b
  fails to hold); or
- a **determinism / append-only violation** — a way the composite key or the migration
  breaks deterministic projection or the append-only/forward-only chain guarantee.

## 7. Test plan (6 tests, TDD — written and shown failing/passing before merge)

1. **Sibling split reproduces #38 and fails pre-change.** Two ballots share a root, declare
   distinct `questionId`s; assert they are *separate* lineages (neither is the other's
   prior). FAILS against current `rootOf`-only grouping (they merge); PASSES after the
   composite key.
2. **Backward-compat for the existing convenings.** A fixture with no `questionId` projects
   byte-identically to the pre-change grouping (the 169 existing convenings unaffected).
3. **Gate not relaxed (NI-16b).** A successor that the CIP-13/CIP-15 gate would reject is
   *still* rejected when a `questionId` is present — the separator can only withhold, never
   grant, a promotion.
4. **NI-13h holds per sibling.** A type-mismatched supersede inside one sibling group is not
   promoted, independent of the separator.
5. **Re-adjudication composes.** A valid in-sibling supersede (same `questionId`) promotes
   the head within its sibling without disturbing the other sibling's head.
6. **Determinism.** Same log + meta → identical sibling lineages across repeated builds
   (ordering stable under the composite key).

---

**Standing lesson:** the merge bug is the mirror image of the supersede gate's purpose. The
gate exists to stop a *weaker* successor from overwriting a head; CIP-16 stops a *different
question* from being mistaken for a successor at all. Both protect the same property — the
record means exactly what it says — from opposite directions. The fix is deliberately the
narrowest possible: an unhashed declared separator that only ever *splits*, verified to be
monotone-downward on promotions so it can never become an edit key in disguise.
