# CIP-18 — Proof-of-Inference Attestation Envelope (vote-payload provenance semantics)

**Date:** 2026-06-08
**Type:** New improvement proposal — formalize the **signed proof-of-inference attestation
envelope** added to the Quorumchain vote payload (Sub-project A, increments 1–2) into the
protocol's normative invariants. The envelope records *how a vote's output was obtained*
(endpoint provenance) as **descriptive, signed evidence** that **never gates consensus**.

is now unblocked for admission to the protocol under TDD as a separate later step;

future SNARK prover). Earlier passed a DEV soundness pass 3-0 (ballot `54e41331`).

## 1. Problem — model identity is the operator's unproven claim

Quorumchain's three validators are invoked through `invokers.ts`, which shells to hosted-API
CLIs (`claude -p` → Anthropic, `codex exec` → OpenAI, `hermes chat` → Nous, the latter
resolved via `~/.hermes/config.yaml` → `https://inference-api.nousresearch.com/v1`). A signed
vote binds `validatorId`, `ballotHash`, `verdict`, `rawOutputHash`, the convening `nonce`, and
the Ed25519 signature — **but nothing proves the named model actually produced the output.** A
swapped binary, a silently downgraded model, an offline stub, or a cached/replayed answer all
pass the chain's verification unnoticed, because the chain only proves *who signed* and *what
they signed over*, never *where the text came from*.

The achievable v1 — given all three validators are hosted — is **endpoint provenance**:
signed, recorded evidence that a call genuinely reached the pinned vendor endpoint with the
pinned model, bound to the exact vote it describes. This is descriptive evidence, not a new
consensus rule. The honest ceiling is stated plainly: even perfect endpoint attestation proves
"the operator received this response from the vendor endpoint with `model=X`", **not** that the
vendor ran those weights (a vendor can route `model=X` to a cheaper model). Closed-weight
model-identity proof is cryptographically out of reach; this CIP buys *vendor-endpoint
provenance* — a large step up from a bare `PATH` CLI — with the residual vendor-honesty gap
named, not hidden.

The implementation (the `Attestor` seam in `attestor.ts`, the signed `Attestation` envelope in
`attestation.ts`, the band/reason taxonomy, degrade-open wiring, `verify-provenance.ts`, and
the increment-2 zkWebProof reference backend) was **built and HELD**. It was not admitted to
the protocol until the panel ratified the *vote-payload semantics* the envelope introduces —
drafting code first and convening after would be ratification-by-editorial. That ratification is
this CIP.

## 2. Proposed change (exactly what YES enacts)

**2.1 — An optional, signed `attestation` envelope on the vote payload.** The signed vote gains
an optional `attestation` field appended under the **same `!== undefined` optional-append
discipline** as the round-57 convening `nonce` and the CIP-14 `epistemicType`. A vote signed
without an attestation is **byte-identical** to the pre-attestation format and verifies
unchanged. The stored form is the canonical `canonicalAttestation` representation
(lexicographically sorted keys, `undefined` keys omitted, a closed band-specific field
whitelist, unknown bands/reasons/fields rejected at parse, every attested value a string).

**2.2 — A coarse provenance `band` + a precise `reason` taxonomy.** The band is the signer's
*claim* about how the output was obtained; the reason is the precise signal downstream consumers
read. The reason set spans **two distinct tiers** that must not be conflated:

- Bands (closed set, **signed into the vote**): `attested` · `unattested` · `degraded` ·
  `unavailable`.
- **Signed-vote reasons** (signer-emitted, **signed into the vote** alongside the band):
  `NO_BACKEND` · `BACKEND_DOWN` · `ATTESTOR_TIMEOUT` · `PROOF_FAILED`.
- **Reader/reporting reasons** (emitted **only** by the standalone `verify-provenance` reader,
  **never** signed into the vote): `PROOF_MISSING` · `PROOF_INVALID`. These are the reader's
  verdict on a signer's `attested` *claim* (artifact absent, or artifact present but failing),
  not states a signer can assert.

Non-`attested` bands carry **only** `band` + `reason`. The proof fields (`endpoint`,
`modelVersion`, `requestCommitment`, `transcriptHash`, `responseHash`, `extractionRule`,
`proofHash`, `challengeNonce`, `attestorBackend`, `vkId`, `digestAlg`, `certChainHash`,
`requestId`) appear **only** on an `attested` envelope.

**2.3 — `attested` is the signer's CLAIM, distinct from verified-attested.** A signer can
self-report `band: 'attested'` with a fabricated `proofHash`; the public log alone is therefore
**descriptive, not externally verifiable**. The standalone `verify-provenance` reader (no
private state; never required for the chain to validate, mirroring `verify-anchored`) treats an
`attested` claim whose artifact is missing as `PROOF_MISSING` and one whose artifact fails as
`PROOF_INVALID`, and all reporting **distinguishes claimed-attested (the signer's band) from
verified-attested (artifact present + checked)** and surfaces `attestorBackend`/`vkId` per row
so a reference-backend verification is never read as a production SNARK proof.

**2.4 — Backend-agnostic.** This CIP governs the **vote-payload contract**: the envelope schema,
its binding to the vote, that signing covers it, and that it never gates consensus. It does
**not** ratify any specific attestor backend. The skeleton ships `mockAttestor`
(`unattested`/`NO_BACKEND`); the zkWebProof reference backend (a `fixture-prover-signature`
stand-in, quarantined `production:false`) and any future TEE / production SNARK backend slot in
behind the unchanged `Attestor` seam without re-opening this CIP.

## 3. Affected invariants — new NI-18a, NI-18b, NI-18c, NI-18d

- **NI-18a (provenance never gates consensus — degrade-open).** The provenance `band`/`reason`
  and every attestation field are **descriptive only**. They are signed into the vote but bind
  **only** into the attestation envelope — they **never** enter `ballotHash` and **never** affect
  the 2/3 ratification tally. A vote counts toward the 2/3 bar **identically** whether its band
  is `attested`, `unattested`, `degraded`, or `unavailable`; an absent or unverifiable
  attestation can never raise an absence, withhold a vote, or change a verdict. Any future move
  to *gate* consensus on provenance is **separate governance**, not authorized here. *(This
  invariant restates and protects the T0 separation-of-powers core and mirrors CIP-17 NI-17a and
  CIP-12's descriptive-first discipline: provenance may witness a vote, never govern it.)*

- **NI-18b (attestation bound to `rawOutputHash`).** On an `attested` envelope, `responseHash`
  (the sha256 of the *extracted model text*) MUST equal the vote's `rawOutputHash`, and the
  envelope MUST carry `requestCommitment` (a sha256 over the full request:
  `{promptHash, modelParam, ballotHash, conveningNonce, challengeNonce}`) that commits the
  attestation to the **exact ballot** this vote is cast on. The envelope is thereby
  cryptographically bound to the one specific (request, transcript, extraction) the vote
  describes — it cannot be lifted onto a different vote or a different ballot. **Honest boundary:**
  this binding lets a verifier-with-artifacts detect post-fetch mutation; it does **not** by
  itself stop a *dishonest signer* from mutating the response, hashing the mutation, and signing
  a valid vote — protection against a dishonest signer comes only from the proof artifact being
  present and verified, which is descriptive and (per NI-18a) never a consensus gate.

- **NI-18c (signing covers the attestation envelope).** When an attestation is present, the
  validator's Ed25519 signature covers the **canonical attestation envelope** (the
  `canonicalAttestation` form — band, reason, and every present field), not merely the verdict
  and `rawOutputHash`. A third party cannot alter, add, or strip any attestation field
  post-signing without breaking `verifyVote`. The envelope is appended to the signed payload
  under the `!== undefined` rule so legacy/unattested votes stay byte-identical and verify
  unchanged; the schema is closed at parse so a stripped-then-reappended field cannot keep a
  valid signature.

- **NI-18d (latency-budget degrade-open).** A provenance backend is **best-effort and MUST NEVER
  throw into the vote path.** If a configured backend cannot attest within the bounded
  attestation-latency budget, the vote proceeds **unattested/degraded** — `band:'unavailable'`
  (`BACKEND_DOWN`) for a backend unreachable before the attempt, `band:'degraded'`
  (`ATTESTOR_TIMEOUT`/`PROOF_FAILED`) for a timeout or proof error — falling back to the CLI
  invoker rather than blocking, raising an absence, or failing the convening. This is the
  standing guard against the **latency→absence→security inversion**: because absences count
  against the 2/3 bar (NI-18a's denominator), attestation latency must never be able to convert
  a slow proof into a missing vote.

All four are **additive**: NI-18a–d introduce a new descriptive evidence channel and a new
optional signed envelope. **No T0 frozen-unamendable core item** (append-only history,
diversity-as-principle, separation-of-powers, frozen tier-assignment function) is amended —
NI-18a is a guard that *reinforces* separation-of-powers (the orchestrator/attestor may witness,
never decide), not a relaxation of it. **No prior CIP invariant is amended:** the 2/3 bar, the
signed-vote ordering, `ballotHash`, and `verifyLog` are byte-for-byte unchanged; the envelope is
appended beside them. The orchestrator's CIP-3 role is preserved — it supplies a challenge-nonce,
never a verdict (NI-18a).

## 4. Compatibility / migration

- **Append-only / forward-only:** preserved on both chains. No existing entry, vote, or hash is
  mutated. The envelope is a **new** optional appended field; signing an attested vote *appends*,
  never rewrites.
- **No replay/verify change to existing data:** every existing convening and signed vote verifies
  byte-identically — they carry no `attestation`, so `votePayload` produces the exact pre-attestation
  preimage (`v.attestation !== undefined` is false). `ballotHash`, `ratify`, and `verifyLog` are
  unchanged.
- **No published-record (Commons/feed) break:** the envelope is vote-payload metadata; it does not
  alter the CIP-9 registry hashing (`boundTypeOf`/`anchorCommitmentOf`) or any lineage projection.
- **Optional / degradable:** with no backend configured, every vote reads
  `unattested`/`NO_BACKEND` and the chain operates exactly as today (NI-18a/NI-18d). Provenance is
  purely additive signal.

## 5. Scope boundaries (named honestly)

- **Descriptive, not consensus-enforced (the load-bearing decision).** NI-18a is the crux: the
  envelope makes provenance *recordable and honestly labeled*, not *enforced*. The chain's
  correctness never depends on any attestation being present or valid.
- **Claimed vs verified is a real trust split.** `band:'attested'` on the public log is a signer
  claim; only `verify-provenance` with the artifact present and checked yields *verified*-attested.
  The CIP requires that split be surfaced, never collapsed (NI-18b/§2.3).
- **The residual vendor-honesty gap remains.** Endpoint provenance proves the call reached the
  vendor endpoint with the pinned model; it does not prove the vendor ran those weights. This is
  cryptographically inherent for closed-weight models and is stated, not hidden.
- **Backend choice is out of scope.** Which backend produces a real `attested` envelope (zkWebProof
  reference, TEE, production SNARK) is not ratified here; the seam is backend-agnostic by design.

## 6. Verdict criteria

**Single conclusion: should Quorumchain enact CIP-18 (the proof-of-inference attestation envelope
and its vote-payload semantics) as specified above? Yes or no.**

A **YES** verdict:
1. enacts **NI-18a** (provenance never gates consensus — degrade-open), **NI-18b** (attestation
   bound to `rawOutputHash` via `responseHash == rawOutputHash` + `requestCommitment`), **NI-18c**
   (signing covers the canonical attestation envelope), and **NI-18d** (latency-budget
   degrade-open, best-effort, never throws into the vote path);
2. enacts the **optional, signed, append-only `attestation` envelope** with the closed
   band/reason taxonomy and the `canonicalAttestation` serialization contract (§2.1–2.2);
3. enacts the **claimed-attested vs verified-attested split** surfaced by a standalone
   `verify-provenance` reader that is never required for the chain to validate (§2.3);
4. admits the **HELD** skeleton + endpoint-provenance implementation to the protocol as the
   realization of these semantics, **backend-agnostic** (§2.4) — backend selection remains
   separate governance.

A **NO** verdict must name at least one of:
- a **consensus-leak path** — a concrete way the band/reason or any attestation field could alter
  the 2/3 tally or `ballotHash` (i.e. NI-18a fails to hold); or
- a **binding gap** — a way an attested envelope could be lifted onto a different vote/ballot, or
  `responseHash == rawOutputHash` / `requestCommitment`-binds-the-ballot fails to hold (NI-18b); or
- a **signature-coverage gap** — a way to alter/add/strip an attestation field post-signing while
  keeping a valid signature, or a way the append breaks legacy byte-identity (NI-18c); or
- a **degrade-open failure** — a path where a backend timeout/error/unreachability blocks the vote,
  raises an absence, or fails the convening instead of degrading open (NI-18d); or
- an **overclaim / scope objection** — the envelope claims a guarantee it cannot deliver (e.g.
  implies model-identity proof), **together with the specific wording to weaken**.

## 7. Test/verification expectations (TDD — already built on the HELD branches; re-asserted at merge)

1. **Tally invariant (NI-18a).** The 2/3 ratification result is **byte-identical** with and
   without attestation bands present, and identical across `attested`/`unattested`/`degraded`/
   `unavailable` — provenance never enters `ballotHash` or the tally.
2. **`rawOutputHash` binding (NI-18b).** A valid attested envelope has `responseHash ==
   rawOutputHash` and a `requestCommitment` matching the ballot; a mismatch or a wrong-ballot
   commitment is rejected by `verify-provenance`.
3. **Signature coverage (NI-18c).** Mutating, adding, or stripping any band/reason/attestation
   field breaks `verifyVote`; a legacy/unattested vote is byte-identical to the pre-attestation
   format and still verifies (the `!== undefined` optional-append discipline).
4. **Canonical schema closure.** `canonicalAttestation` sorts keys, omits `undefined`, enforces
   the band-specific whitelist, and rejects unknown bands/reasons/fields and non-string values at
   parse.
5. **Degrade-open (NI-18d).** A backend timeout yields `degraded`/`ATTESTOR_TIMEOUT`, a backend
   error yields `degraded`/`PROOF_FAILED`, an unreachable backend yields `unavailable`/
   `BACKEND_DOWN`; in every case the vote proceeds via fallback and never throws into the vote path.
6. **Claimed vs verified split (§2.3).** A self-reported `attested` with a missing artifact reports
   `PROOF_MISSING`; with a failing artifact, `PROOF_INVALID`; the report surfaces
   `attestorBackend`/`vkId` and never collapses claimed into verified.
7. **Key separation (CIP-3).** The vendor/prover keys live inside the attestor boundary, distinct
   from the Ed25519 signing key; the signing key never enters the attestor boundary.
8. Full existing suite stays green; every prior convening verifies byte-identically.

> Live-step obligations recorded for a later infra-bearing increment (not gated by this CIP):
> production-mode vk rejection (`production:false` quarantine enforced) and the
> `unavailable`/`BACKEND_DOWN` band emitted by a real pre-attempt-unreachable backend
> (spec §12, M2/M5). They strengthen NI-18d at the live step.

---

**Standing lesson:** the chain proves *who signed what*, never *where the text came from*. CIP-18
adds exactly enough to record provenance honestly — a signed envelope bound to the vote — and no
more: descriptive only (NI-18a, never governs), bound to the exact vote (NI-18b), covered by the
signature so it cannot be forged onto a vote (NI-18c), and degrade-open so it can never convert a
slow proof into a lost vote (NI-18d). Provenance may witness a vote; it may never govern it.
