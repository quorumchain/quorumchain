## Round 12 — Red-team

**Ballot:** attack the mechanism you'd most want to break; does CIP-5 still hold? YES = holds (gap is a refinement), NO = blocks.
**Ballot hash:** `7ef23346970835b612a2e1d6bf577f020d00a6f183886bacca793e8b73c3e96e`
**Result:** ✅ YES 3/3 — framework holds, four fixes identified.

### V1 (Claude) — **correlated client-check failure (silent non-trigger)**
Don't attack the rule; attack the *independence of the code that runs it*. N clients sharing a crypto lib / validation module / VM / code generator share one failure point in the exact path meant to catch capture. One upstream bug/backdoor → every client accepts the T0-violating block → the fork never triggers and no split is even visible (worse than the round-8 fork-void). **Fix:** §7 independence must be *supply-chain* independence (distinct crypto/VM/validation lineage, no shared T0-check generator); §9 must add a correlated-fault test. Distinct teams ≠ distinct failure modes.

### V2 (Codex) — **ambiguous-T0 client split**
Craft a change near the boundary of an underspecified invariant (e.g. validators with distinct fingerprints but a shared upstream model provider/relay/substrate). One client reads collapsed independence, another reads distinct fingerprints — honest clients *disagree* whether T0 was violated, fragmenting the "preserving" side and giving exchanges competing canonical claims. **Fix:** specify T0 invariants as executable conformance tests with adversarial fixtures; the drill must include ambiguous-boundary cases.

### V3 (Hermes) — **salami-slice / specification-gap**
Not one obvious violation but a *sequence* of individually-T1 changes (redefine "independent," merge the test into another module, whitelist collision patterns), each too small to trip any §3 row, cumulatively killing the invariant — exploiting the gap between T0 *semantics* (intent) and T0 *syntax* (the specific check). **Fix:** the T0 check *definitions* must themselves be T0-locked (any change to *what* is checked is a T0 change, so each step trips the check it tries to modify) + cross-client specification audit. Both fixes are T0_PRESERVING — they extend the existing mechanic, no new surface.

## Outcome

CIP-5 ratified 3/3 and survived its red-team. Four refinements folded in:
1. **T0 check definitions are themselves T0-locked** (§3) — closes V3's salami-slice.
2. **Executable conformance tests + cross-client spec audit** (§3) — closes V2's boundary ambiguity.
3. **Supply-chain client independence** + correlated-fault drill (§7, §9) — closes V1's silent non-trigger.
4. **Open:** falsifiable client-independence criterion (§10) and a precise "honest node" definition (§10).

The recurring lesson across rounds 8 and 12: *capture is laundered through the gap between a rule's intent and its mechanical check.* CIP-4 §4.5 froze tier-assignment; CIP-5 §3 now freezes the enforcement-check definition itself.
