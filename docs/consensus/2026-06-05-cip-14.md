# CIP-14 — Type-Bound Ballot Hash (signed)

**Date:** 2026-06-05
**Subject:** [[CIP-14]] — bind a claim's `epistemicType` **into** the ballot hash (and thus
into every validator signature), closing the [[CIP-13]] NI-13a ideal that "type is SIGNED,

legacy hashes stay byte-identical.

## 1. Problem

[[CIP-13]] NI-13a stated the ideal that a claim's `epistemicType` is **frozen and signed,
never inferred** — but at v0.1 the type rode along as **declared meta**, outside the hashed
commitment. Validators signed the prompt + context, not the type. So the type was not
actually part of what each validator vouched for, and a type could in principle be relabelled
without breaking any signature. This is a change to the **trust root** (what validators sign
over), and was red-teamed as adversarially as the CIP-10 seed construction.

## 2. Spec (exactly what was ratified)

- **Optional-append hashing.** `ballotHash(prompt, context)` stays **byte-identical** (v1,
  legacy). `ballotHash(prompt, context, boundType) = sha256(JSON.stringify({prompt, context,
  epistemicType}))` (v2). The signer hashes the type too, so it enters **every** validator
  signature — the type becomes **SIGNED**, not merely declared.
- **`meta.typeBinding='hashed'`** is the per-entry discriminator a verifier reads to recompute
  *with or without* the type — **no version integer**, no ambiguity a forger could exploit to
  present a v2 type-bound claim as v1 (or vice-versa) to launder a type.
- **Only `epistemicType` is bound** (YAGNI — no speculative extra fields).
- **Forward-only.** The 61 backfilled advisory types are **not** re-convened; no lineage break.

## 3. Invariants introduced — NI-14a … NI-14f

- **NI-14a** — **legacy byte-identical.** A v1 ballot's hash is unchanged by the new path;
  the optional-append never shifts an existing hash.
- **NI-14b** — relabelling an advisory (unhashed) type does **not** silently become a binding
  one; binding requires the `typeBinding='hashed'` declaration.
- **NI-14d** — binding is the **proposer's signed declaration**, **orthogonal** to the v0.3
  panel *ratification* of a type (the [[CIP-13]] type sub-claim) — both signed layers coexist;
  one does not stand in for the other.
- **NI-14f** — `meta.typeBinding='hashed'` is set **iff** the type was actually hashed; it is
  the verifier's recompute discriminator and is enforced at hash-construction time (the type is
  in `boundTypeOf`, deliberately NOT something a later edit can fake).

*(Gates G14a..f and the TDD plan are in the CIP-14 spec; the ratified consequence is that a
type change is now a supersession, not a silent edit — strengthening NI-13d.)*

## 4. Compatibility / migration

Append-only / forward-only preserved: legacy hashes are byte-identical (NI-14a), so every
already-signed ballot still passes `verifyEntry`; new type-bound ballots are additive. The
change touches the trust root (what is hashed/signed) but adds **no** T0 frozen-unamendable
core violation — the signed hash-chained vote model itself ([[CIP-3]]) is preserved and
extended, not amended away.
