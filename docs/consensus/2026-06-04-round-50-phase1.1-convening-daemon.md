## V2's REVISE — a real crash-recovery laundering gap (fixed under TDD)

V1 and V3 each audited all five invariants mechanically and passed them. Both **also
noticed the exact spot V2 flagged** — V1 judged it "biases toward finalizing not
refishing," V3 logged it as a "noted design trade-off (not a REVISE-level gap)." V2
alone called it disqualifying, and on the project's own standard (the gap between a
rule's *intent* and its mechanical *check* is where capture hides) V2 was right to:

> "There is a real no-laundering gap in `queue.complete()`: it writes `done/<id>.json`
> and then unlinks `pending/<id>.json`. A crash between those operations leaves the same
> decided ballot in both `done/` and `pending/`, and `listPending()` will reprocess it
> on the next daemon run… That violates the invariant that a quorum-participated
> outcome, including a genuine NO, is final and never retried."

The original comment in `queue.ts` *bragged* about this as "the safe direction" — the
classic intent-vs-check slip: re-convening a decided ballot is not safety, it is exactly
the laundering invariant 2 forbids (re-run a genuine NO until it flips).

### The fix (TDD, red→green)
- **`listPending` now excludes any id that already has a terminal (`done/` or `failed/`)
  record**, even if a `pending/` file for it still exists. A ballot decided before a
  crash interrupted cleanup is never re-listed, so it is never re-convened. The `done/`
  record is authoritative.
- Two crash-recovery tests (re-create the stale `pending` file after `complete`/`fail`,
  assert `listPending` returns `[]`). **Suite 171 → 173, all green.**
- **Verified on the live queue, zero inference cost:** re-created a stale `pending` file
  for the round-50 ballot (already in `done/`) and ran the daemon → `Done: [] Retried: []
  Failed: []`, log unchanged at 141 entries, chain valid. A decided ballot is now final
  even across a crash.

## What the panel affirmed (invariants 1–5)

- **1 Custody** — `liveRunBallot` only spawns hosts, pin-checks, convenes, closes; it
  loads only public pinned keys and supplies no verdict. Each host signs behind the
  Signer/process boundary; `ratify` recomputes from signed votes. (CIP-3, carried.)
- **2 No laundering** — the decided/retry split is `votes.length >= quorum`
  (*participation*, not YES-count): a fully-participated NO goes to `done/` and is never
  retried. (The crash-recovery hole in this invariant is what V2 closed.)
- **3 Liveness ≠ verdict** — a throw (no host up / key substituted) or sub-quorum
  participation is a retry (`bumpAttempt`, then `failed/` after `maxAttempts`); no vote
  is ever fabricated, no liveness blip finalized as a NO.
- **4 Pin/absence** — `assertMatchesPin` rejects substitution/unknown, allows absence;
  absence counts against ratify's 2/3-of-registered-panel bar. (Phase 0.2/0.5, carried.)
- **5 Termination** — `drainQueue` snapshots `listPending()` and iterates the snapshot;
  bumped ballots are not reprocessed in-pass. A single drain always terminates.
