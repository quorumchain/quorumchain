# Ratify CIP-14 (Type-Bound Ballot Hash), closing the NI-13a ideal. PROPOSAL: bind epistemicType into ballotHash using the SAME optional-append discipline as the round-57 nonce. ballotHash(prompt,context) stays byte-identical (v1, legacy); ballotHash(prompt,context,boundType)=sha256(JSON.stringify({prompt,context,epistemicType})) (v2). A per-entry flag meta.typeBinding='hashed' is the discriminator a verifier reads to recompute with-or-without the type (no version integer). The signer hashes the type too, so it enters every validator signature — the type becomes SIGNED, not merely declared. Consequences: (1) a type change becomes a supersession not a silent edit (NI-13d); (2) binding is the proposer's signed DECLARATION, orthogonal to v0.3 panel RATIFICATION — both signed layers coexist; (3) only epistemicType is bound (YAGNI); (4) the 61 backfilled advisory types are NOT re-convened (forward-only, no lineage break). Invariants NI-14a..e, gates G14a..f, TDD plan in the spec. Is this design sound to implement, or does it have a flaw that warrants rejection/amendment?

### Contrary anchors (each clears the symmetric anchor bar)

- **documented-cryptographic-attack-class / standards best-current-practice** — IETF RFC 8725 (JSON Web Token Best Current Practices), §2.1/§3.1 'alg' header validation; CVE-2015-9235 / CVE-2016-5431 class (JWT algorithm-confusion, alg=none, RS256→HS256 substitution): contradicts _Attack surface (a): that an unsigned per-entry discriminator (meta.typeBinding='hashed') selecting the verification preimage is safe. RFC 8725 documents that letting an attacker-controllable, out-of-band field choose how a verifier processes a signed object is a recurring root cause of forgery/laundering. CIP-14's flag is NOT covered by any signature; its safety rests entirely on every verification path RE-deriving the preimage from (prompt,context,boundTypeOf(meta)) and rejecting on hash mismatch (G14c/G14d). The anchor's residual bite: any present-or-future verifier that trusts vote.ballotHash or verifyVote alone WITHOUT recomputing the expected hash from the flag reintroduces the exact alg-confusion footgun — type-laundering becomes possible. The design is sound; the unsigned discriminator shifts a load-bearing invariant from cryptography onto caller discipline._

### Searched, rejected (suppression audit-trail, NI-AA8)

- RFC 8785 (JSON Canonicalization Scheme) / known non-determinism of JSON.stringify (insertion-order key emission, cross-language preimage divergence): rejected — Real concern for hash-over-JSON schemes, but PRE-EXISTING in the v1 ballotHash, not introduced by CIP-14. CIP-14 appends epistemicType as a fixed trailing key under the same serialization the system already depends on (signed-vote.ts:71-73); it neither worsens nor relies on new canonicalization assumptions. Does not contradict CIP-14 specifically — it contradicts the whole pre-existing hashing convention, out of scope for this ratification.
- TLS/SSL version-downgrade & rollback attacks (e.g., POODLE, FREAK); generic protocol-downgrade literature: rejected — The v2→v1 'strip the type' downgrade was the obvious analog to check. The CIP-14 design defeats it by reconstruction cross-check: stripping meta.typeBinding makes the verifier recompute the v1 hash, which mismatches the signed v2 ballotHash → reject (G14c/G14d, fail-closed). No fail-open downgrade path exists in the design as specified; SHA-256 second-preimage resistance blocks a colliding v1 preimage. Anchor does not bite.
- CIP-14 token-set injection (malformed/unrecognized epistemicType smuggled into the v2 preimage): rejected — Checked the chokepoint: ballotHash re-validates against BOUND_TYPES at the SINGLE place a type can enter the hash (signed-vote.ts:53,72); an unrecognized token with the flag set falls through to v1 byte-identically. NI-14f serialization contract is enforced at the crypto core, not at the caller, so it cannot be bypassed by a malformed type. No injection surface.
- CIP-15 anchorCommitment composition (a SECOND optional-append added after epistemicType): rejected — Examined as a potential collision/ambiguity introduced by stacking optionals. It is CONFIRMING, not contrary: each of {none, type-only, anchor-only, both} yields a distinct preimage by distinct keys, empty-args stays byte-identical to v1. The design's optional-append discipline composed cleanly to a second field without collision — evidence the v1-byte-identical invariant is robust, not fragile. (Also post-dates CIP-14 ratification; not chargeable against it regardless.)
- Attack surfaces (b) re-typing rigidity / (c) validator vouches for proposer-chosen type: rejected — Searched for an external principle that metadata corrections must not require supersession, or that signing over 'what I was asked (incl. type)' over-extends a validator's vouch. Found none with external-anchor standing. The spec internally resolves both: NI-13d models type change as supersession-with-retained-lineage (consistent with append-only commons), and ballotHash binds 'what the validator was asked' (CIP-1 1c bait-and-switch defense) — the type is part of the question posed, not a claim of independent type-adjudication; consequence-(2) keeps declaration orthogonal to ratification. These are design-judgment positions; contrary view is unanchored opinion and does not clear the bar.

### Falsification conditions (what anchored evidence would warrant re-adjudication)

- toward **MATERIAL**: A concrete verification path in the specified design (ratify/verifyEntry/read-surface, or a documented verifier contract) that consumes meta.typeBinding to display or accept a bound type WITHOUT recomputing the expected hash from (prompt,context,boundTypeOf(meta)) and comparing — i.e., a fail-OPEN read where flipping the unsigned flag changes the accepted type without a hash mismatch. That would convert the RFC 8725 alg-confusion anchor from a discipline-residual into a live laundering vector.
- toward **MATERIAL**: A demonstrated serialization divergence specific to the type-append: an input where JSON.stringify of {prompt,context,epistemicType:T} and {prompt,context} (or two distinct bound types) collide or where a legacy v1 entry's bytes shift when the v2 code path runs — falsifying NI-14a's byte-identical guarantee in the actual implementation rather than only in v1's pre-existing assumptions.
- toward **DECISIVE**: A cryptographic result (peer-reviewed or CVE-grade) showing the optional-append-without-domain-separator construction admits a second-preimage / cross-version collision under SHA-256 — i.e., a (prompt',context') whose v1 hash equals some (prompt,context,type) v2 hash — which would let a forger present a type-bound signed ballot as untyped (or vice-versa) while passing the reconstruction cross-check. No such result exists for SHA-256; absent it, the downgrade stays fail-closed.
