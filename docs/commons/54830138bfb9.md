### Searched, rejected (suppression audit-trail, NI-AA8)

- RFC 3161 Time-Stamp Protocol / OpenTimestamps / Chainlink OCR round-time / Pyth VAA publish-time: rejected — Checked as a 'soundness' anchor (cryptographic timestamping exists, so surfacedAt is fine). It cures nothing: the spec leaves surfacedAt self-declared and does NOT require it to equal any of these crypto timestamps for a credited anchor. The standard's existence is evidence FOR the flaw (verifiable time requires crypto proof, not assertion), confirming surface (a) rather than refuting it. primary-document/standard
- RFC 8785 JSON Canonicalization Scheme (JCS): rejected — Checked as 'canonicalization is a solved problem, so surface (c) is moot.' A solution the spec does not mandate does not make the spec sound. sha256(canonical(anchors)) is unspecified re: JCS, duplicate contentHashes, and unicode/whitespace in free-text citedAssertion; and a SECOND CIP-14 optional-append reintroduces the exact null/undefined serialization ambiguity CIP-14 had to fold. Availability of JCS confirms the gap is real and fixable in-spec — i.e., should be folded pre-code. primary-document/standard
- W3C Verifiable Credentials Data Model + Data Integrity (signature/issuerKey-based provenance): rejected — Checked as 'signed issuer + issuerKey gives trustworthy provenance, mooting surface (b).' VC trust requires a resolvable trust registry / DID resolution — an ONLINE/registry dependency. NI-15g demands VERIFIED provenance OFFLINE; offline a declared upstream provenanceTrace is generally unverifiable, so requiredFamilies>=2 is frequently unreachable and the NI-15g(anti-gaming) vs NI-11g(liveness floor) tension is real and unresolved. Anchor confirms the tension, does not dissolve it. standard
- Spec's own honest 'content-pending' label on structural-only promotion (surfaces d/e): rejected — Checked as 'honest labeling makes structural-only promotion sound by design.' Honest labeling is necessary but not sufficient: promotable() returning true on a structurally-valid-but-irrelevant anchor can flip the lineage head / reorder the Commons render / dequeue reviewQueue BEFORE content verification runs — observable harm and an NI-15b violation if any downstream consumer reads promoted-but-pending as resolved. The label does not prevent the side effect, so it does not establish soundness. internal-spec-artifact
- NI-15e anchorCommitment = sha256(canonical(anchors)) bound via CIP-14 optional-append (the genuinely sound part): rejected — Checked whether the commitment-binding being cryptographically sound rebuts the NO verdict. It is sound as a swap-prevention mechanism, but it does not contradict 'a flaw warrants amendment' — the flaws are on surfaces (a)/(c)/(b)/(d), independent of the commitment binding. Soundness of one mechanism is not evidence the spec as a whole needs no amendment. internal-spec-artifact
